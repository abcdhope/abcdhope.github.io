GeeRPC

本章开始学习7天用Go从零实现RPC框架GeeRPC

RPC（Remote Procedure Call，远程过程调用）是一种计算机通信协议，允许调用不同进程空间的程序。也就是说一个程序在机器A，而另一个程序在机器B，当A的程序想要调用B的程序时，需要通过网络来传输数据。

RPC框架需要解决的问题

  • 通信方式

    RPC首先要解决的是两个程序之间的通信方式，如果这个两个应用程序位于不同的机器，那么一般会选择 TCP 协议或者 HTTP 协议；那如果两个应用程序位于相同的机器，也可以选择 Unix Socket 协议。

  • 编码格式

    当两个程序在传输数据的过程中需要对数据进行编码，那么就需要确定报文的编码格式，最常用的是JSON或XML，如果报文过大，也可以选择protobuf等编码格式。

  • 注册中心(registry)和负载均衡(load balance)

    当服务端的实例很多时，客户端需要选择一个实例进行通信来获取结果。这时服务端可以在启动时将自己注册到注册中心，客户端在注册中心选择一个进行通信，这样服务端和客户端只需要感知注册中心的存在而不需要感知对方的位置。注册中心通常还需要实现服务动态添加、删除，使用心跳确保服务处于可用状态等功能。

  • 公共能力

    假设服务端是由不同的团队提供，RPC框架可以提供公共的能力来为不同的服务使用，例如消息编解码、连接池、收发线程、超时处理等技术，避免了重复劳动。

Day1 服务端与消息编码

本章的主要目标是：

  • 使用 encoding/gob 实现消息的编解码(序列化与反序列化)
  • 实现一个简易的服务端，仅接受消息，不处理。

消息的序列化与反序列化（编码与解码）

一个典型的RPC调用如下所示：

err = client.Call("Arith.Multiply", args, &reply)

其中，客户端发送的请求包括服务名（结构名）Arith、方法名Multiply、参数args这三个，服务端返回错误error、返回值reply这两个。

将客户端和服务端的参数和返回值抽象为body，剩下的信息（服务名、方法名、错误）存入结构Header中，因此Header的结构如下：

//Header，存放客户端请求的服务、方法以及服务端返回的错误
type Header struct{
	ServiceMethod string//例如Arith.Multiply
	Seq uint64//请求序号，区分不同的请求
	Error string//错误信息，客户端为空，服务端若有错误则置入其中
}

由于存在不同的编解码方式（gob、json），为了能够使用不同的编解码方式，抽象出对消息体进行编解码的接口 Codec，该接口中定义编解码函数、链接关闭函数：

//为了能够调用不同的编解码函数，抽象它们的函数
type Codec interface {
	io.Closer                         //关闭链接方法
	ReadHeader(*Header) error         //解码Header内容
	ReadBody(interface{}) error       //解码Body内容
	Write(*Header, interface{}) error //对Header和Body编码
}

GOb结构

定义Gob结构，结构中包含请求链接、链接的缓冲、编码对象、解码对象：

//定义gob结构体，里面存放请求链接、编码、解码、缓冲
type GobCodec struct {
	conn io.ReadWriteCloser //请求链接
	buf  *bufio.Writer      //防止阻塞而创建的带缓冲的 Writer
	dec  *gob.Decoder       //解码
	enc  *gob.Encoder       //编码
}

定义构造函数：

//构造函数
func NewGobCodec(conn io.ReadWriteCloser) Codec {
	buf := bufio.NewWriter(conn)
	return &GobCodec{
		conn: conn,
		buf:  buf,
		dec:  gob.NewDecoder(conn),
		enc:  gob.NewEncoder(buf),
	}
}

定义4个函数以实现接口:

func (c *GobCodec) ReadHeader(h *Header) error {
	return c.dec.Decode(h)
}

func (c *GobCodec) ReadBody(body interface{}) error {
	return c.dec.Decode(body)
}

func (c *GobCodec) Write(h *Header, body interface{}) (err error) {
	defer func() {
		_ = c.buf.Flush() //清除缓冲
		if err != nil {
			_ = c.Close()
		}
	}()
	//编码
	//对header编码
	if err := c.enc.Encode(h); err != nil {
		log.Println("rpc codec: gob error encoding header:", err)
		return err
	}
	//对body编码
	if err := c.enc.Encode(body); err != nil {
		log.Println("rpc codec: gob error encoding body:", err)
		return err
	}
	return nil
}

//关闭链接
func (c *GobCodec) Close() error {
	return c.conn.Close()
}

通信过程

在http报文中分为header和body两部分，header中的content-Type和Content-Length指定body的格式和长度，服务端通过解析 header 就能够知道如何从 body 中读取需要的信息，在RPC协议中，这部分内容是自己定义的，一般情况下，在报文的最开始会规划固定的字节来协商相关的信息。例如：第1个字节用来表示序列化方式，第2个字节表示压缩方式，第3-6字节表示 header 的长度，7-10 字节表示 body 的长度。

在geerpc中，需要确定的是序列化方式，由于本章只定义了Gob，因此序列化方式只有一种。除了确定序列化方式，还需要确定报文发送方是谁，因此定义一个常量来标志这个报文来自客户端：

//定义配置信息，即客户端的传输方式
//定义客户端请求标志
const MagicNumber = 0x3bef5c

//定义结构，包含请求标志和解码方式
type Option struct {
	MagicNumber int
	CodeType    codec.Type
}

//初始化
var DefaultOption = &Option{
	MagicNumber: MagicNumber,
	CodeType:    codec.GobType,
}

GeeRPC 客户端固定采用 JSON 编码 Option，后续的 header 和 body 的编码方式由 Option 中的 CodeType 指定，服务端首先使用 JSON 解码 Option，然后通过 Option 的 CodeType 解码剩余的内容，因此报文的内容格式为：

| Option{MagicNumber: xxx, CodecType: xxx} | Header{ServiceMethod ...} | Body interface{} |
| <------      固定 JSON 编码      ------>  | <-------   编码方式由 CodeType 决定   ------->|

由于传输的内容不止一个，因此Header和body可能不止一个：

| Option | Header1 | Body1 | Header2 | Body2 | ...

服务端

利用net.Listen函数来开启监听，利用Accept函数来接收客户端的请求，然后开启goroutine来并行启动业务。

//定义服务端
type Server struct{}

//返回空结构
func NewServer() *Server { return &Server{} }

//定义接收方法
func (server *Server) Accept(lis net.Listener) {
	for {
		conn, err := lis.Accept()
		if err != nil {
			log.Println("rpc server: accept error:", err)
			return
		}
		go server.ServeConn(conn) //处理请求，即报文内容
	}
}

定义一个默认的服务端对象，并定义一个函数来调用Accept方法，这样的目的是方便调用：

//定义默认服务端对象
var DefaultServer = NewServer()

func Accept(lis net.Listener) { DefaultServer.Accept(lis) }

现在的重点就是ServeConn方法，该方法就是对报文的处理，在本章的作用是按照刚才的设定来编解码报文内容：首先是读取头部内容，根据头部信息来选择编解码类型：

//对头部解码，并根据读取到的Option来选取剩下的报文内容的解码方式
func (server *Server) ServeConn(conn io.ReadWriteCloser) {
	defer func() {
		_ = conn.Close()
	}()
	//定义变量来存放Option
	var option Option
	//解码
	if err := json.NewDecoder(conn).Decode(&option); err != nil {
		log.Println("rpc server: 头部的配置信息出现错误: ", err)
	}
	//判断请求标志是否正确
	if option.MagicNumber != MagicNumber {
		log.Printf("rpc server: 无效的请求标志 %x", option.MagicNumber)
		return
	}
	//根据读取的codecType来选择构造函数
	f := codec.NewCodecFuncMap[option.CodecType]
	if f == nil {
		log.Printf("rpc server: 无效的编解码类型 %s", option.CodecType)
		return
	}
	//开始对报文内容解码
	server.serveCodec(f(conn))//输入为Codec对象
}

选择了编解码方式后，定义方法serveCodec实现对报文内容的处理。

报文内容的处理分为三个阶段：

  • 读取请求报文
  • 处理请求报文
  • 回复请求

由于在一次连接中有许多请求，因此需要无限制的等待请求，直到错误（连接关闭，或报文有问题）。虽然处理请求可以并发进行，但回复请求需要逐个发送，因为并发发送会导致多个报文交织在一起，因此需要用锁来保证发送顺序。

func (server *Server) serveCodec(cc codec.Codec) {
	//定义锁
	sending := new(sync.Mutex)
	wg := new(sync.WaitGroup)
	//读取报文、处理报文、回复报文
	for {
		//读取报文
		req, err := server.readRequest(cc)
		if err != nil {
			//如果读取的报文为空，则停止循环
			if req == nil {
				break
			}
			//将错误信息添加至结构中
			req.h.Error = err.Error()
			//回复
			server.sendResponse(cc, req.h, req.replyv.Interface(), sending)
		}
		wg.Add(1)
		//处理报文，处理结束后调用回复方法
		go server.handleRequest(cc, req, sending, wg)
	}
    //等待所有协程结束
	wg.Wait()
	_ = cc.Close()
}

其中读取、处理、回复三大方法如下所示：

//定义读取流程、处理流程、回复流程
//定义结构体，用于存放读取的内容
type request struct {
	h            *codec.Header //结构名、方法名、请求序列号、错误信息
	argv, replyv reflect.Value //方法参数、回复内容
}

func (server *Server) readRequestHeader(cc codec.Codec) (*codec.Header, error) {
	var h codec.Header
	//将解码的内容写入到h中
	if err := cc.ReadHeader(&h); err != nil {
		if err != io.EOF && err != io.ErrUnexpectedEOF {
			log.Println("rpc server: 读取Header错误: ", err)
		}
		return nil, err
	}
	return &h, nil
}

func (server *Server) readRequest(cc codec.Codec) (*request, error) {
	//读取header
	h, err := server.readRequestHeader(cc)
	if err != nil {
		return nil, err
	}
	//定义request对象，存储内容
	req := &request{h: h}
	// TODO: now we don't know the type of request argv
	// day 1, just suppose it's string
	req.argv = reflect.New(reflect.TypeOf(""))
	if err = cc.ReadBody(req.argv.Interface()); err != nil {
		log.Println("rpc server: 读取方法参数错误: ", err)
		return nil, err
	}
	return req, nil
}

func (server *Server) sendResponse(cc codec.Codec, header *codec.Header, body interface{}, sending *sync.Mutex) {
	sending.Lock()
	defer sending.Unlock()
	//编码
	if err := cc.Write(header, body); err != nil {
		log.Println("rpc server: 回复出现错误: ", err)
	}
}

func (server *Server) handleRequest(cc codec.Codec, req *request, sending *sync.Mutex, wg *sync.WaitGroup) {
	// TODO, should call registered rpc methods to get the right replyv
	// day 1, just print argv and send a hello message
	defer wg.Done()
	//打印结构方法、参数
	log.Println(req.h, req.argv.Elem())
	//仅回复该请求的序号
	req.replyv = reflect.ValueOf(fmt.Sprintf("geerpc resp %d", req.h.Seq))
	//回复
	server.sendResponse(cc, req.h, req.replyv.Interface(), sending)
}

定义一个结构体，用于存放读取的信息，即Header、body，readRequest包含了对Header的读取，由于不知道主体内容的数据类型，这里假设为string类型，且调用reflect库的对象来读取内容。

读取内容之后，对内容进行处理，这里只是输出请求的序列号，通过sendResponse发送给客户端。发送的方式为调用codec的函数Write来编码header和新body（即序列号）。

func startServer(addr chan string) {
	//开启监听端口
	l, err := net.Listen("tcp", ":0")
	if err != nil {
		log.Fatal("network error:", err)
	}
	log.Println("start rpc server on", l.Addr())
	//将地址发送
	addr <- l.Addr().String()
	geerpc.Accept(l)
}

func main() {
	addr := make(chan string)
	go startServer(addr)

	//实现客户端
	conn, _ := net.Dial("tcp", <-addr)

	defer func() {
		_ = conn.Close()
	}()

	time.Sleep(time.Second)
	//编码头部信息
	_ = json.NewEncoder(conn).Encode(geerpc.DefaultOption)
	//定义编解码
	cc := codec.NewGobCodec(conn)

	//发送消息和接收消息
	for i := 0; i < 5; i++ {
		//定义header
		h := &codec.Header{
			ServiceMethod: "Foo.Sum",
			Seq:           uint64(i),
		}
		_ = cc.Write(h, fmt.Sprintf("geerpc req %d", h.Seq))
		//读取消息
		_ = cc.ReadHeader(h)
		var reply string
		_ = cc.ReadBody(&reply)
		log.Println("reply:", reply)
	}
}

定义一个chan string类型的addr，addr负责接收地址，然后将地址发给客户端。服务端的开启由一个函数来实现，主要是调用net.Listen来开启监听，然后Accept函数来开始处理请求。

Day2 高性能客户端

+　实现一个支持异步和并发的高性能客户端

Call的设计

Call的作用在于存放传送的信息，包括方法名、相关参数以及服务器返回的信息，并且设定一个通道用于判断是否传送完成，若传送完成则定义的对象接收通知：

//定义调用一次RPC所需要的信息
type Call struct {
	Seq           uint64      //请求序号
	ServiceMethod string      //服务方法名字
	Args          interface{} //方法参数
	Reply         interface{} //回复内容
	Error         error       //返回的错误信息
	Done          chan *Call  //请求处理完成后，通知调用方,以便于异步调用
}

func (call *Call) done() {
	call.Done <- call
}

客户端

客户端结构中包含以下内容：

  • 编解码器，即自定义的Codec，输入为链接，得到编解码器；
  • 互斥锁，该锁的作用为在客户端发送请求时防止报文混淆以及在其他方面例如关闭链接；
  • Header文件，其文件里是方法名、序号、服务器返回的错误信息；
  • 请求编号，用于确定请求信息的序号；
  • 链接关闭信号，该信号分为两种，一种是服务器关闭链接的信号，一种是客户端关闭链接的信号；
  • 存放未处理完的请求的哈希表，键为编号，值为Call对象，当请求完成后从表中删除。

针对以上需求，构造客户端结构为：

//客户端
type Client struct {
	cc       codec.Codec      //链接，用于编解码
	opt      *Option          //编解码配置
	sending  sync.Mutex       //发送中的锁
	mu       sync.Mutex       //平时用的锁
	header   *codec.Header    //header信息,包括客户端请求的服务、方法以及服务端返回的错误
	seq      uint64           //请求序号
	pending  map[uint64]*Call //存放未完成的请求
	closing  bool             //客户端主动终止链接
	shutdown bool             //服务器终止链接
}

并且定义链接关闭方法以及判定链接是否关闭的方法：

//定义shutdown终止信息
var ErrShutDown = errors.New("链接被服务器关闭")

//关闭链接
func (client *Client) Close() error {
	client.mu.Lock()
	defer client.mu.Unlock()
	if client.shutdown {
		return ErrShutDown
	}
	//关闭链接
	client.closing = true
	return client.cc.Close()
}

//判断链接是否存活
func (client *Client) IsAvailable() bool {
	client.mu.Lock()
	defer client.mu.Unlock()
	return !client.closing && !client.shutdown
}

定义变量来测试该结构是否符合io.closer接口的类型：

var _ io.Closer = (*Client)(nil)

下面定义注册call方法以及从哈希表中删除call的方法和当链接关闭时对call对象的处理方法：

//定义关于Call的方法，即注册到哈希表中、删除表、以及链接关闭时将其他正在处理的请求标志为完成状态
//注册请求
func (client *Client) registerCall(call *Call) (uint64, error) {
	client.mu.Lock()
	defer client.mu.Unlock()
	if client.closing || client.shutdown {
		return 0, ErrShutDown
	}
	//赋予编号
	call.Seq = client.seq
	client.pending[client.seq] = call
	//增大编号
	client.seq++
	return call.Seq, nil
}

//删除请求
func (client *Client) removeCall(seq uint64) *Call {
	client.mu.Lock()
	defer client.mu.Unlock()
	call := client.pending[seq]
	delete(client.pending, seq)
	return call
}

//当链接关闭时调用
func (client *Client) terminateCall(err error) {
	//防止继续将请求发送给服务端
	client.sending.Lock()
	defer client.sending.Unlock()
	//防止对请求进行操作
	client.mu.Lock()
	defer client.mu.Unlock()
	//输入错误信息以及完成标志
	for _, call := range client.pending {
		call.done()
		call.Error = err
	}
}

客户端接收响应

接收响应存在三种情况：

  • call不存在，但是服务端还是在处理请求；
  • call存在，但是服务端在处理请求时出现错误；
  • call存在，服务端正常处理请求并返回。

针对这三种情况，可采用switch标识符来进行判断，第一种判断的条件为哈希表中是否存有该call,不存在则返回读取空内容发生的错误信息；第二种情况为判断传回来的header对象中error是否为空,不为空则该错误置入call对象中，并且生成一个读取空内容的错误；第三种就是默认情况，读取返回的内容,若读取出现错误则将该错误存入call对象中。

//接收服务端返回的信息
func (client *Client) receive() {
	var err error
	//一直接收，直到出错
	for err == nil {
		//读取header
		var h codec.Header
		if err = client.cc.ReadHeader(&h); err != nil {
			break
		}
		//找到目标call
		call := client.removeCall(h.Seq)
		switch {
		case call == nil:
			err = client.cc.ReadBody(nil)
		case h.Error != "":
			//转换为错误对象
			call.Error = fmt.Errorf(h.Error) //将错误存入到Call之后呢？
			err = client.cc.ReadBody(nil)
			call.done()
		default:
			err = client.cc.ReadBody(call.Reply)
			if err != nil {
				call.Error = errors.New("读取内容错误：" + err.Error())
			}
			call.done()
		}
	}
	//通知其他未完成的请求并将shutdown置为true，防止其他请求也收到同样地错误
	client.terminateCall(err)
}

	client.sending.Lock()
	defer client.sending.Unlock()

	//注册call
	seq, err := client.registerCall(call)
	if err != nil {
		call.Error = err
		call.done()
		return
	}
	//定义header
	client.header.ServiceMethod = call.ServiceMethod
	client.header.Seq = seq
	client.header.Error = ""
	//发送信息
	if err = client.cc.Write(&client.header, call.Args); err != nil {
		call := client.removeCall(seq)
		//call可能为nil，这通常意味着写入部分失败。客户端已经收到响应并处理了
		if call != nil {
			call.Error = err
			call.done()
		}
	}
}

		Done:          done,

对 net/rpc 而言，一个函数需要能够被远程调用，需要满足如下五个条件：

  • the method’s type is exported. – 方法所属类型是导出的。
  • the method is exported. – 方法是导出的。
  • the method has two arguments, both exported (or builtin) types. – 两个入参，均为导出或内置类型。
  • the method’s second argument is a pointer. – 第二个入参必须是一个指针，原因为第二个参数为返回参数，在服务端进行处理时需要将值写进参数中，这时客户端可以读取该值。
  • the method has return type error. – 返回值为 error 类型。

例如：

func (t *T) MethodName(argType T1, replyType *T2) error

通过反射实现service

定义方法结构体，结构体内部存放方法和相关参数：

type methodType struct {
	method    reflect.Method
	ArgType   reflect.Type
	ReplyType reflect.Type
	numCalls  uint64
}

定义初始化方法，用于提取存放的参数：

func (m *methodType) NumCalls() uint64 {
	return atomic.LoadUint64(&m.numCalls) //返回加载到*addr的值
}

//定义Arg
func (m *methodType) newArgv() reflect.Value {
	var argv reflect.Value
	//指针类型
	if m.ArgType..Kind() == reflect.Ptr {
		argv = reflect.New(m.ArgType.Elem())
	} else {
		argv = reflect.New(m.ArgType).Elem() //值类型转换为指针指向的值
	}
	return argv
}

//定义ReplyType
func (m *methodType) newReplyv() reflect.Value {
	// reply must be a pointer type
	replyv := reflect.New(m.ReplyType.Elem())
	//查找
	switch m.ReplyType.Elem().Kind() {
	case reflect.Map:
		replyv.Elem().Set(reflect.MakeMap(m.ReplyType.Elem()))
	case reflect.Slice:
		replyv.Elem().Set(reflect.MakeSlice(m.ReplyType.Elem(), 0, 0))
	}
	return replyv
}

定义结构体service，使得该结构体能够解析输入的结构体对象，并寻找满足条件的方法，此处的条件为：输入参数的个数为3个，输出参数为1个，且输入参数为结构体对象rcvr、方法的参数Args、返回的参数reply，输出的参数为error:

type service struct {
	name   string                 //结构体名称
	typ    reflect.Type           //结构体类型
	rcvr   reflect.Value          //结构体对象
	method map[string]*methodType //映射的结构体的所有符合条件的方法。
}

func newService(rcvr interface{}) *service {
	s := new(service)
	s.rcvr = reflect.ValueOf(rcvr)
	s.name = reflect.Indirect(s.rcvr).Type().Name()
	s.typ = reflect.TypeOf(rcvr)
	if !ast.IsExported(s.name) {
		log.Fatalf("rpc server: %s is not a valid service name", s.name)
	}

	return s
}

func (s *service) registerMethods() {
	s.method = make(map[string]*methodType)
	//寻找满足条件的方法
	for i := 0; i < s.typ.NumMethod(); i++ {
		method := s.typ.Method(i) //第i个方法
		mType := method.Type
		//如果入参的个数不为3(第一个入参为结构体对象本身)或输出参数的个数不为1
		if mType.NumIn() != 3 || mType.NumOut() != 1 {
			continue
		}
		//如果
		if mType.Out(0) != reflect.TypeOf((*error)(nil)).Elem() {
			continue
		}
		argType, replyType := mType.In(1), mType.In(2)
		//判断类型是否导出
		if !isExportedOrBuiltinType(argType) || !isExportedOrBuiltinType(replyType) {
			continue
		}
		s.method[method.Name] = &methodType{
			method:    method,
			ArgType:   argType,
			ReplyType: replyType,
		}
		log.Printf("rpc server: register %s.%s\n", s.name, method.Name)
	}
}

func isExportedOrBuiltinType(t reflect.Type) bool {
	return ast.IsExported(t.Name()) || t.PkgPath() == ""
}

增加调用方法：

//调用方法
func (s *service) call(m *methodType, argv, replyv reflect.Value) error {
	//增加调用次数
	atomic.AddUint64(&m.numCalls, 1)
	f := m.method.Func
	//利用反射调用方法
	returnValue := f.Call([]reflect.Value{s.rcvr, argv, replyv})
	if errInter := returnValue[0].Interface(); errInter != nil {

}

func (server *Server) findService(serviceMethod string) (svc *service, mtype *methodType, err error) {
	dot := strings.LastIndex(serviceMethod, ".")
	if dot < 0 {
		err = errors.New("rpc server: service/method request ill-formed: " + serviceMethod)
		return
	}
	//读取方法名
	serviceName, methodName := serviceMethod[:dot], serviceMethod[dot+1:]
	//从Map中读取结构
	svci, ok := server.serviceMap.Load(serviceName)
	if !ok {
		err = errors.New("rpc server: 没有在找到该服务: " + serviceName)
	}
	svc = svci.(*service)
	mtype = svc.method[methodName]
	if mtype == nil {
		err = errors.New("rpc server: 没有找到该方法" + methodName)
	}
	return
}

func (server *Server) readRequest(cc codec.Codec) (*request, error) {
	//读取header
	h, err := server.readRequestHeader(cc)
	if err != nil {
		return nil, err
	}
	//定义request对象，存储内容
	req := &request{h: h}
	req.svc, req.mtype, err = server.findService(h.ServiceMethod)
	if err != nil {
		return req, err
	}
	req.argv = req.mtype.newArgv()
	req.replyv = req.mtype.newReplyv()

	//判断req.argv是否为指针
	argvi := req.argv.Interface()
	if req.argv.Type().Kind() != reflect.Ptr {
		argvi = req.argv.Addr().Interface()
	}
	//传入argvi
	if err = cc.ReadBody(argvi); err != nil {
		log.Println("rpc server: 读取方法参数错误: ", err)
		return nil, err
	}
	return req, nil
}

Day4 超时处理

在整个远程调用的过程中，客户端和服务端都有需要超时处理的地方：

  • 客户端
    • 连接服务端超时
    • 在发送请求到服务端之前，写报文导致超时
    • 等待服务端处理时导致超时
    • 接收响应后，读报文导致超时
  • 服务端
    • 读取客户端请求报文时，超时
    • 发送响应报文时，写报文超时
    • 调用映射服务的方法时，处理报文导致超时

针对上述情况，在下列几处添加超时处理机制：

  • 客户端连接服务端
  • 客户端调用Call方法（此处包括写报文、等待服务端处理、读报文，因为这些操作都发生在调用Call方法后）
  • 服务端处理报文

创建连接超时

在配置Option中添加连接时间限制和处理报文的时间限制，并在默认配置中添加默认时间：

type Option struct {
	MagicNumber    int           // MagicNumber marks this's a geerpc request
	CodecType      codec.Type    // client may choose different Codec to encode body
	ConnectTimeout time.Duration // 连接限制时间,0表示没有限制
	HandleTimeout  time.Duration
}

在客户端创建链接处和创建客户端处添加超时处理：创建链接的函数net.Dial改为能够进行超时处理的函数net.DialTimeout；针对创建客户端的超时处理，通过通道channel和select来进行超时处理：

//存放客户端，用于通道传送
type clientResult struct {
	client *Client
	err    error
}
type newClientFunc func(conn net.Conn, opt *Option) (client *Client, err error)

func dialTimeout(f newClientFunc, network string, address string, opts ...*Option) (client *Client, err error) {
	opt, err := parseOption(opts...)
	if err != nil {
		return nil, err
	}
	//创建链接,DialTimeout限制了链接时间，超过时间则表示连接失败
	conn, err := net.DialTimeout(network, address, opt.ConnectTimeout)
	if err != nil {
		return nil, err
	}
	//延迟关闭链接，如果创建的客户端为空
	defer func() {
		//如果创建客户端的时候出错，则关闭链接
		if err != nil {
			_ = conn.Close()
		}
	}()
	//利用通道来创建客户端
	ch := make(chan clientResult)
	go func() {
		client, err := f(conn, opt)
		ch <- clientResult{client: client, err: err}
	}()
	//如果ConnectTimeout==0，则不需要超时处理
	if opt.ConnectTimeout == 0 {
		result := <-ch
		return result.client, result.err
	}

	select {
	case <-time.After(opt.ConnectTimeout):
		return nil, fmt.Errorf("rpc client: 连接超时: 希望连接时间范围在%d", opt.ConnectTimeout)
	case result := <-ch:
		return result.client, result.err
	}

}

//创建链接，并创建客户端
func Dial(network string, address string, opts ...*Option) (client *Client, err error) {
	return dialTimeout(NewClient, network, address, opts...)
}

Client.Call 超时

利用context包来实现Client.Call方法的超时处理机制:

//封装Go，用于阻塞call.Done，等待响应返回
func (client *Client) Call(ctx context.Context, serviceMethod string, args, reply interface{}) error {
	call := client.Go(serviceMethod, args, reply, make(chan *Call, 1))
	select {
		//通过设置的超时时间来完成事件
	case <-ctx.Done():

		return errors.New("rpc client: call failed: " + ctx.Err().Error())
	case call := <-call.Done:
		return call.Error
	}
}

服务端超时处理

服务端在处理报文的时候利用select和time.After添加超时处理。

创建两个通道：called 和sent ，前者用于完成调用方法之后，而后者是完成发送报文之后。这两个通道与客户端的通道定义不同，后者由于需要返回客户端对象，因此定义了一个含有客户端对象的结构体，而前者不需要返回信息，因此通道里定义一个空结构体来表示完成操作：

func (server *Server) handleRequest(cc codec.Codec, req *request, sending *sync.Mutex, wg *sync.WaitGroup, timeout time.Duration) {
	// TODO, should call registered rpc methods to get the right replyv
	// day 1, just print argv and send a hello message
	defer wg.Done()
	//定义通道
	called := make(chan struct{})
	sent := make(chan struct{})
	go func() {
		err := req.svc.call(req.mtype, req.argv, req.replyv)
		called <- struct{}{}
		if err != nil {
			req.h.Error = err.Error()
			server.sendResponse(cc, req.h, invalidRequest, sending)
			sent <- struct{}{}
		}
		//回复
		server.sendResponse(cc, req.h, req.replyv.Interface(), sending)
		sent <- struct{}{}
	}()

	//没有定义超时限制
	if timeout == 0 {
		<-called
		<-sent
		return
	}

	select {
	case <-time.After(timeout):
		req.h.Error = fmt.Sprintf("prc server: 请求处理超时: 处理时间范围应在 %s", timeout)
		server.sendResponse(cc, req.h, invalidRequest, sending)
	case <-called:
		<-sent
	}
}

Day5 支持HTTP协议

在WEB中，通常利用HTTP协议的几个关键字如HEAD、GET、POST来发送请求，但是在rpc协议中它的消息格式与HTTP协议的消息格式并不兼容，针对此问题，HTTP协议的CONNECT方法充当代理服务来中转消息。

假设浏览器与服务器的HTTPS通信都是加密的，而浏览器请求的站点地址和端口被加密保存在HTTPS请求报文中，那么代理服务器该如何知道请求的站点和端口？

解决方案是：浏览器先通过HTTP协议明文传输站点地址和端口到代理服务器中，代理服务器接收到请求后，会在对应的站点和端口建立TCP连接，连接成功后返回HTTP 200状态码来告知浏览器与服务器的加密通道连接成功，之后浏览器可以通过代理服务器来传输HTTPS报文，而代理服务器仅将报文转发到服务器中。

例如：

  1. 浏览器向代理服务器发送CONNECT请求：
CONNECT geektutu.com:443 HTTP/1.0
  1. 代理服务器返回200状态码表示连接已经建立：
HTTP/1.0 200 Connection Established
  1. 之后浏览器和服务器开始 HTTPS 握手并交换加密数据，代理服务器只负责传输彼此的数据包，并不能读取具体数据内容（代理服务器也可以选择安装可信根证书解密 HTTPS 报文）。

从上述过程得知，这个过程的作用是通过代理服务器将HTTP协议转换成HTTPS协议。而对于RPC服务端来说，需要将HTTP协议转换成RPC协议，对于客户端来说，需要通过HTTP CONNECT请求创建连接。

服务端支持HTTP协议

RPC的通信过程为：

  1. 客户端向RPC服务器发送CONNECT请求：

    CONNECT 10.0.0.1:9999/_geerpc_ HTTP/1.0

  2. RPC服务器返回HTTP 200状态码表示连接建立

    HTTP/1.0 200 Connected to Gee RPC

  3. 客户端使用创建好的连接发送RPC报文，先发送Option，再发送N个请求报文，服务器处理RPC请求并响应。

让服务端支持HTTP协议，使得客户端能够向服务器发送CONNECT请求来创建请求链接，链接创建成功后，客户端使用创建好的连接发送 RPC 报文，先发送 Option，再发送 N 个请求报文，服务端处理 RPC 请求并响应。

建立一个API来选择创建链接方式：

// XDial calls different functions to connect to a RPC server
// according the first parameter rpcAddr.
// rpcAddr is a general format (protocol@addr) to represent a rpc server
// eg, http@10.0.0.1:7001, tcp@10.0.0.1:9999, unix@/tmp/geerpc.sock
func XDial(rpcAddr string, opts ...*Option) (*Client, error) {
	parts := strings.Split(rpcAddr, "@")
	if len(parts) != 2 {
		return nil, fmt.Errorf("rpc client err: wrong format '%s', expect protocol@addr", rpcAddr)
	}
	protocol, address := parts[0], parts[1]
	switch protocol {
	case "http":
		return DialHTTP("tcp", address, opts...)
	default:
		return Dial(protocol, address, opts...)
	}
}

Day6 负载均衡

负载均衡策略包括下列几种：

  • 随机选择策略 - 从服务列表中随机选择一个。
  • 轮询算法(Round Robin) - 依次调度不同的服务器，每次调度执行 i = (i + 1) mode n。
  • 加权轮询(Weight Round Robin) - 在轮询算法的基础上，为每个服务实例设置一个权重，高性能的机器赋予更高的权重，也可以根据服务实例的当前的负载情况做动态的调整，例如考虑最近5分钟部署服务器的 CPU、内存消耗情况。
  • 哈希/一致性哈希策略 - 依据请求的某些特征，计算一个 hash 值，根据 hash 值将请求发送到对应的机器。一致性 hash 还可以解决服务实例动态添加情况下，调度抖动的问题。一致性哈希的一个典型应用场景是分布式缓存服务。这个策略在分布式缓存中得到实现。

在本节中，利用随机选择和轮询算法来实现RPC的负载均衡策略。

服务发现

RPC的负载均衡策略的前提在于有多个具有相同功能的服务实例来供客户端选择。

建立一个具有同一功能的服务中心，用于存放服务以及通过负载均衡策略获取服务。定义一个接口，用于实现最基本的功能，例如更新服务中心里的服务列表（从其它服务器中更新服务、手动更新服务）、获取服务中心里的服务（目标服务以及全部服务），接口内容定义为：

// MultiServersDiscovery is a discovery for multi servers without a registry center
// user provides the server addresses explicitly instead
type MultiServersDiscovery struct {
	r       *rand.Rand //随机选择的序号
	mu      sync.RWMutex

	index   int //轮询算法的序号
}

func NewMultiServersDiscovery(servers []string) *MultiServersDiscovery {
	d := &MultiServersDiscovery{
		r:       rand.New(rand.NewSource(time.Now().Unix())),//由时间戳定义的随机种子
		servers: servers,
	}

	return nil

//更新服务列表
func (d *MultiServersDiscovery) Update(servers []string) error {
	d.mu.Lock()
	defer d.mu.Unlock()
	d.servers = servers
	return nil
}

func (d *MultiServersDiscovery) Get(mode SelectMode) (string, error) {
	d.mu.Lock()
	defer d.mu.Unlock()
	n := len(d.servers)
	if n == 0 {
		return "", errors.New("服务列表中的服务个数为0")
	}
	switch mode {
	case RandomSelect:
		return d.servers[d.r.Intn(n)], nil
	case RoundRobinSelect:
		//轮询算法，从初始位置逐个加1
		s := d.servers[d.index%n] //d.index可能会超过n，因为d.index的初始值是随机生成的
		d.index = (d.index + 1) % n
		return s, nil
	default:
		return "", errors.New("rpc discovery: 没有支持的服务选择模式")
	}
}

func (d *MultiServersDiscovery) GetAll() ([]string, error) {
	//防止获取的途中服务列表进行更新
	d.mu.Lock()
	defer d.mu.Unlock()
	s := make([]string, len(d.servers))
	copy(s, d.servers) //避免修改s导致d.servers被改变
	return s, nil
}

接下来定义一个方法，用于客户端请求所有可用的服务，即调用负载均衡策略中GetAll的方法来获取所有服务，然后并发计算结果并返回其中一个：

//将请求广播给所有服务,调用所有服务来获得结果
func (xc *XClient) BroadCast(ctx context.Context, serviceMethod string, args, reply interface{}) error {
	//获取所有服务
	servers, err := xc.d.GetAll()
	if err != nil {
		return err
	}
	//定义锁
	var wg sync.WaitGroup
	var mu sync.Mutex
	var e error
	replyDone := reply == nil
	ctx, cancel := context.WithCancel(ctx)
	for _, rpcAddr := range servers {
		wg.Add(1)
		go func(rpcAddr string) {
			defer wg.Done()
			var clonedReply interface{}
			//将reply的内容复制
			if reply != nil {
				clonedReply = reflect.New(reflect.ValueOf(reply).Elem().Type()).Interface()
			}
			err := xc.call(rpcAddr, ctx, serviceMethod, args, clonedReply)
			mu.Lock()
			//如果任意一个服务调用出现错误，则快速结束
			if err != nil && e == nil {
				e = err
				cancel()
			}
			//将clonedReply的值赋值给reply
            //存疑!replyDone
			if err == nil && !replyDone {
				//指针赋值
				reflect.ValueOf(reply).Elem().Set(reflect.ValueOf(clonedReply).Elem())
				replyDone = true
			}
			mu.Unlock()
		}(rpcAddr)
	}
	wg.Wait()
	return e
}

Day7 服务发现与注册中心

  • 实现一个简单的注册中心，支持服务注册、接收心跳等功能
  • 客户端实现基于注册中心的服务发现机制

注册中心的位置

注册中心的位置如下图所示：

其功能表现为：

  • 服务端启动后，向注册中心发送注册消息，注册中心得知该服务已经启动，且处于可用状态。在服务端启动还需要定期向注册中心发送心跳来表示该服务端还在正常运作。
  • 客户端向注册中心发送询问信息，注册中心向客户端返回可用的服务列表。
  • 客户端选择一个可用的服务来完成事务。

注册中心的功能并不局限于以上，还可以包括配置的动态同步、通知机制等。比较常用的注册中心有etcd、zookeeper、consul等。

Gee Registry

本节的注册中心实现的功能主要是支持心跳保活。因此需要设置一个时间来保证一个服务能够存活多长时间，且需要一个map来存储服务以及它的注册时间。因此定义两个结构体，一个用来设置时间和存储服务，另一个用来描述服务和注册时间：

type GeeRegisty struct {
	timeout time.Duration
	mu      sync.Mutex
	servers map[string]*ServerItem
}

type ServerItem struct {
	Addr  string
	start time.Time
}

ServerItem结构用来存放服务的地址以及它的注册时间。

随后进行初始化：

func New(timeout time.Duration) *GeeRegisty {
	return &GeeRegisty{
		timeout: timeout,
		servers: make(map[string]*ServerItem),
	}
}

设置默认的心跳保活时间和服务访问地址并设置默认对象：

const (
	defaultPath    = "/_geerpc_/registry"
	defaultTimeout = time.Minute * 5 //默认心跳保活5分钟
)
var DefaultGeeRegistry = New(defaultTimeout)

之后定义两个方法：添加服务到列表putServer以及获取可用服务列表aliveServers:

//添加可用服务
func (r *GeeRegisty) putServer(Addr string) {
	r.mu.Lock()
	defer r.mu.Unlock()
	s := r.servers[Addr]
	//若注册中心没有该服务，则注册服务
	if s == nil {
		r.servers[Addr] = &ServerItem{Addr: Addr, start: time.Now()}
	} else {
		//存在则更新注册时间
		s.start = time.Now()
	}
}

//返回可用服务
func (r *GeeRegisty) aliveServers() []string {
	r.mu.Lock()
	defer r.mu.Unlock()
	var alive []string
	for addr, s := range r.servers {
		//判断服务存在的时间是否超过保活的时间范围
		if r.timeout == 0 || s.start.Add(r.timeout).After(time.Now()) {
			alive = append(alive, addr)
		} else {
			delete(r.servers, addr)
		}
	}
	//对服务排序
	sort.Strings(alive)
	return alive
}

putServer方法负责将服务添加到列表中，并重置注册时间；aliveServers方法负责检查列表中的服务的存在时间是否超过规定的保活时间，即s.start.Add(r.timeout).After(time.Now())，此段代码的含义是注册时间+保活时间是否大于目前的时间，若大于则服务还在保活时间范围内返回true，反之返回false。

外部向注册中心添加服务和获取服务的方式是通过HTTP协议完成的，因此定义ServeHTTP方法来返回内容：

func (r *GeeRegisty) ServeHTTP(w http.ResponseWriter, req *http.Request) {
	switch req.Method {
	case "GET":
		//消息头，由键值对组成，通过键名可获取对应的值
		w.Header().Set("X-Geerpc-Servers", strings.Join(r.aliveServers(), ","))
	case "POST":
		addr := req.Header.Get("X-Geerpc-Servers") //从消息头中获取信息
		if addr == "" {
			w.WriteHeader(http.StatusInternalServerError)
			return
		}
		r.putServer(addr) //将服务添加到列表中
	default:
		w.WriteHeader(http.StatusMethodNotAllowed)
	}
}

//开启ServeHTTP
func (r *GeeRegisty) HandleHTTP(registryPath string) {
	http.Handle(registryPath, r)
	log.Println("rpc registry path:", registryPath)
}

//默认的服务获取的地址位置
func HandleHTTP() {
	DefaultGeeRegistry.HandleHTTP(defaultPath)
}

针对不同的请求方法来作出对应的措施：GET请求方法调用aliveServers方法来获取可用的服务列表并写入消息头中进行返回；POST请求方法从消息头中获取服务，并利用putServer方法将服务添加至列表中。”X-Geerpc-Servers”为自定义的消息头名称。

心跳保活

定义函数来将目标服务地址添加至列表中，然后定时更新注册时间，这里设置了4分钟来更新注册时间，防止服务被列表删除。

//服务器向注册中心发送心跳来保证服务存活
func sendHeartbeat(registry, addr string) error {
	log.Println(addr, "send heart to registry", registry)
	httpClient := &http.Client{}
	//创建POST请求，目标地址为registry
	req, _ := http.NewRequest("POST", registry, nil)
	//设置请求内容，即更新服务注册信息来确保存活
	req.Header.Set("X-Geerpc-Servers", addr)
	//发送请求
	if _, err := httpClient.Do(req); err != nil {
		log.Println("rpc server: heart beat err:", err)
		return err
	}
	return nil
}

//定时发送心跳
func HeartBeat(registry, addr string, duration time.Duration) {
	if duration == 0 {
		//默认为4分钟，因为要确保在服务被注册中心删除之前更新注册时间
		duration = defaultTimeout - time.Duration(1)*time.Minute
	}
	var err error
	err = sendHeartbeat(registry, addr) //更新注册时间
	go func() {
		//每隔duration的时长发送一次心跳
		t := time.NewTicker(duration)
		for err == nil {
			<-t.C
			err = sendHeartbeat(registry, addr)
		}
	}()
}

HeartBeat设置默认时间为4分钟，并设置计时器来定时更新注册时间。sendHeartbeat函数负责定义请求信息，利用http.NewRequest定义POST请求，创建请求对象http.Client，利用httpClient.Do将请求发送给注册中心。

客户端获取服务

定义结构体来继承之前定义的负载均衡策略，继承的好处是可以直接调用父类的变量属性，并通过父类的服务列表作为客户端需要的服务列表，然后再从中选择一个服务来完成事务。除此之外，新的结构体内部还定义了服务列表更新，设置时间来获取新的服务列表并存储。

type GeeRegistryDiscovery struct {
	*MultiServersDiscovery
	registry   string        //注册中心的地址
	timeout    time.Duration //服务列表的过期时间
	lastUpdate time.Time     //最后从注册中心更新服务列表的时间，默认 10s 过期，即 10s 之后，需要从注册中心更新新的列表。
}

const (
	defaultUpdateTimeout = 10 * time.Second//默认的列表更新时间
)

func NewGeeRegistryDiscovery(registryAddr string, timeout time.Duration) *GeeRegistryDiscovery {
	if timeout == 0 {
		timeout = defaultUpdateTimeout
	}
	return &GeeRegistryDiscovery{
		MultiServersDiscovery: NewMultiServersDiscovery(make([]string, 0)),
		registry:              registryAddr,
		timeout:               timeout,
	}
}

重定义4个方法：Update, Refresh, Get, GetAll：

//更新服务列表
func (d *GeeRegistryDiscovery) Update(servers []string) error {
	d.mu.Lock()
	defer d.mu.Unlock()
	d.servers = servers       //更新服务列表
	d.lastUpdate = time.Now() //重置服务列表更新时间
	return nil
}

func (d *GeeRegistryDiscovery) Refresh() error {
	d.mu.Lock()
	defer d.mu.Unlock()
	//比较过期时间
	if d.lastUpdate.Add(d.timeout).After(time.Now()) {
		return nil
	}
	log.Println("rpc registry: refresh servers from registry", d.registry)
	resp, err := http.Get(d.registry)
	if err != nil {
		log.Println("rpc registry refresh err:", err)
		return err
	}
	servers := strings.Split(resp.Header.Get("X-Geerpc-Servers"), ",")
	d.servers = make([]string, 0, len(servers))
	//获取服务列表
	for _, server := range servers {
		if strings.TrimSpace(server) != "" {
			d.servers = append(d.servers, strings.TrimSpace(server))
		}
	}
	//更新服务时间
	d.lastUpdate = time.Now()
	return nil
}

func (d *GeeRegistryDiscovery) Get(mode SelectMode) (string, error) {
	if err := d.Refresh(); err != nil {
		return "", err
	}
	return d.MultiServersDiscovery.Get(mode)
}

func (d *GeeRegistryDiscovery) GetAll() ([]string, error) {
	if err := d.Refresh(); err != nil {
		return nil, err
	}
	return d.MultiServersDiscovery.GetAll()
}

Update方法负责更新服务列表，并重置最后更新时间；Refresh方法负责查看服务列表的更新时间是否失效，如果失效则从注册中心获取服务列表并重置最后更新时间，通过http.Get方法向注册中心进行请求，随后读取消息头中的服务列表并存入自己的服务列表中。Get和GetAll方法中新增刷新服务列表的操作，然后获取服务。
