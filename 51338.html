分布式缓存GeeCache

该文档是对7天用Go从零实现分布式缓存GeeCache教程系列的总结。

Day1 LRU缓存淘汰策略

假定缓存能够使用的内存为N，当某一时间点增加一条缓存记录时超过了N，这时就需要删除一些缓存来腾空位置。那么删除哪些缓存较为合适？这里有三种缓存淘汰策略：FIFO(First In First Out)，LFU(Least Frequently Used)和 LRU(Least Recently Used)

缓存淘汰策略

FIFO

顾名思义，先进来的缓存先被淘汰。这时就会出现一些问题，经常访问的数据会被频繁地添加进缓存，又被淘汰出去，导致缓存命中率降低。

LFU

该策略认为，最少使用的缓存即为被淘汰的缓存。因此该策略需要维护一个按照访问次序排序的队列，当访问某条缓存时，该条缓存的访问次序加1，然后该队列重新排序。这样的问题就是需要维护该队列使得内存占用高，且该策略受限于历史缓存，即某条缓存以前访问量大而现在很少访问，但是它的访问次数还是很高，这就导致该条缓存迟迟不能被淘汰。

LRU

该策略认为最近最少使用的缓存应该被淘汰。这个策略是FIFO和LFU的结合体，既限制了时间又限制了访问次数，当最近一段时间某条缓存的访问量最少则被淘汰。算法的实现为建立一条队列，当访问某条缓存时就将该缓存移至最后，而最前的缓存则被淘汰。

LRU实现

数据结构

LRU算法的主要数据结构由队列和字典组成。字典存储键和值的映射关系，作用为查询缓存；而队列有值组成，当访问某条缓存时该条缓存移动至末尾。如下图所示LRU算法的主要数据结构：

因此根据该图我们需要创建一个包括字典和队列的结构体Cache，以便增删改查。

结构体的定义如下所示：

/*
定义缓存结构体，包括：
最大内存maxBytes
当前内存nbytes
链表（队列）ll，利用标准库list实现
字典cache
删除操作OnEvicted，当删除某条内存时的一些操作
*/
type Cache struct {
	maxBytes int64                    //最大缓存数
	nBytes   int64                    //当前缓存数
	ll       *list.List               //链表
	cache    map[string]*list.Element //键值对，用于查询对应的值
	//删除操作后的回调函数
	OnEvicted func(key string, value Value)
}

list.Element为链表中的元素，可从链表中获取。

list.Element中的Value为空接口，可被结构体Entry所赋值，Entry里保存键和值：

//保存键值对
type Entry struct {
	key   string
	value Value
}

Value为接口，其内为：

type Value interface {
	Len() int //计算值的大小
}

Len()函数用于计算数据值的大小。

初始化函数为：

//初始化
func New(maxBytes int64, onEvicted func(key string, value Value)) *Cache {
	return &Cache{
		maxBytes:  maxBytes,
		ll:        list.New(),
		cache:     make(map[string]*list.Element),
		OnEvicted: onEvicted,
	}
}

为了方便统计链表中元素的个数，为Cache结构体定义一个方法以便外部调用：

//定义Len方法，用于统计链表中元素的个数
func (c *Cache) Len() int {
	return c.ll.Len()
}

增删改查

1、增改操作被定义在一个函数中：

//增加/修改记录操作
func (c *Cache) Add(key string, value Value) {
	//当链表中有该记录，则进行刷新
	if ele, ok := c.cache[key]; ok {
		//将记录移至第一位
		c.ll.MoveToFront(ele)
		//修改链表元素对应的Value
		//Value为空接口，可用于接收Entry结构，结构中包含对应的键和value值
		kv := ele.Value.(*Entry) //此操作为类型断言，并将
		//修改内存
		c.nBytes += int64(value.Len()) - int64(kv.value.Len())
		//修改字典中的值
		kv.value = value
	} else {
		//在链表中创建元素，并将元素推到第一位
		//输入为元素值value，为空接口类型
		ele := c.ll.PushFront(&Entry{key, value})
		//将元素存放至字典中
		c.cache[key] = ele
		//增加内存
		c.nBytes += int64(len(key)) + int64(value.Len())
	}

	//循环判断内存是否超过最大内存
	for c.maxBytes != 0 && c.maxBytes < c.nBytes {
		c.RemoveOldest() //处理最不常用的记录
	}
}

增改的思路为：

  • 判断链表中是否存在这个元素
    • 若存在，则将该元素移动至首位，并且更新使用内存和字典中键对应的值
    • 若不存在，则在链表中创建一个元素，输入为Entry结构体对象，然后在字典中添加键和对应的值，增加使用内存。
  • 判断使用内存是否超过最大内存，超过则调用删除函数

2、删除函数被定义为删除链表中末尾的元素（经常使用的元素会被调用到首位）：

//删除最旧的记录操作
func (c *Cache) RemoveOldest() {
	//获取末尾的元素
	ele := c.ll.Back()
	if ele != nil {
		//从链表中删除元素
		c.ll.Remove(ele)
		//从元素中获取对应的结构体值
		kv := ele.Value.(*Entry)
		//从字典中删除对应的键值对
		delete(c.cache, kv.key)
		//减少使用内存
		c.nBytes -= int64(len(kv.key)) + int64(kv.value.Len())
		//调用回调函数
		if c.OnEvicted != nil {
			c.OnEvicted(kv.key, kv.value)
		}
	}
}

3、通过键名来查询对应的值：

//查询操作
func (c *Cache) Get(key string) (value Value, ok bool) {
	//调用字典查询对应的值
	if ele, ok := c.cache[key]; ok {
		//从Entry结构体中调取对应的值
		kv := ele.Value.(*Entry)
		return kv.value, true
	}
	return
}

Day2 单机并发缓存

sync.Mutex互斥锁

为了实现LRU缓存的并发控制，采用互斥锁来进行处理。为了保证多个协程访问同一变量时出现冲突，利用互斥锁来给不同的协程加锁。

以下面的例子来解释互斥锁：

var set = make(map[int]bool, 0)
var m sync.Mutex

func printOnce(num int) {
	m.Lock()
	defer m.Unlock()
	if _, exist := set[num]; !exist {
		fmt.Println(num)
	}
	set[num] = true
}

func main() {
	for i := 0; i < 10; i++ {
		go printOnce(100)
	}
	time.Sleep(time.Second)
}
输出为：
100

该函数的作用为若字典中的key没出现过则打印，后续不再打印。锁的作用在于其他协程只能等待用锁的协程进行解锁后才能使用，这时字典已经存在该key了，因此其他协程不再打印。

并发读写

在上一章中，我们实现了lru的基本功能，即对数据增删改查。而在实现之后数据类型为接口类型，需要实现Len()方法才能传入。为了能够支持任意的数据类型的存储，定义一个结构类型来将数据类型全部转换为字节数组，并且这个结构实现Len()方法来统计数据的长度，并且提供String方法来转换成字符类型：

type ByteView struct {
	b []byte
}

//长度
func (v ByteView) Len() int {
	return len(v.b)
}

//字节数组转换为字符串
func (v ByteView) String() string {
	return string(v.b)
}

//复制方法
func (v ByteView) ByteSlice() []byte {
	return cloneBytes(v.b)
}

//克隆
func cloneBytes(b []byte) []byte {
	c := make([]byte, len(b))
	copy(c, b) //复制
	return c
}

定义了数据类型之后，就需要对LRU进行封装，并提供锁功能防止并发冲突，主要功能有：增加数据和获取数据：

/*
封装lru方法
以Value接口实现的值被ByteView进行封装
*/

type cache struct {
	mu         sync.Mutex //锁
	lru        *lru.Cache //链表
	cacheBytes int64      //最大缓存
}

func (c *cache) add(key string, value ByteView) {
	c.mu.Lock()
	defer c.mu.Unlock()
	//如果没有定义缓存链表，则初始化
	if c.lru == nil {
		c.lru = lru.New(c.cacheBytes, nil)
	}
	c.lru.Add(key, value)
}

func (c *cache) get(key string) (value ByteView, ok bool) {
	c.mu.Lock()
	defer c.mu.Unlock()
	//判断链表是否为空
	if c.lru == nil {
		return
	}

	//查询
	if v, ok := c.lru.Get(key); ok {
		return v.(ByteView), ok //v返回的是接口，因此需要进行转换
	}
	return
}

主体结构Group

当实现了增加和获取功能后就可以实现一些需求：

负责与外部交互，控制缓存存储和获取的主流程:

                            是
接收 key --> 检查是否被缓存 -----> 返回缓存值 ⑴
                |  否                         是
                |-----> 是否应当从远程节点获取 -----> 与远程节点交互 --> 返回缓存值 ⑵
                            |  否
                            |-----> 调用`回调函数`，获取值并添加到缓存 --> 返回缓存值 ⑶

当输入key后进行判定，是否存在这个缓存，若存在则调取该缓存，否则从其他地方进行获取。本节只实现了回调函数这一个功能，远程节点留到后面进行编写，因此这里通过调用回调函数来获取缓存。

本节实现一个分组，每个分组里存放着不同类型的数据，例如缓存学生的成绩命名为 scores，缓存学生信息的命名为 info，缓存学生课程的命名为 courses。

定义一个结构体，结构体属性包括分组名称，缓存处理结构，回调函数。回调函数通过实现接口型函数来获取，接口型函数的实现过程如下：

//定义回调接口函数
type GetterFunc func(key string) ([]byte, error)

type Getter interface {
	Get(key string) ([]byte, error)
}

func (g GetterFunc) Get(key string) ([]byte, error) {
	return g(key)
}

这样定义的好处是匿名函数可以直接进行调用，结构体也可以通过实现接口来调用Get函数。这个用法在很多地方都有使用，例如：

在http包中通过实现Handler接口来调用函数：

type Handler interface {
	ServeHTTP(ResponseWriter, *Request)
}
type HandlerFunc func(ResponseWriter, *Request)

func (f HandlerFunc) ServeHTTP(w ResponseWriter, r *Request) {
	f(w, r)
}

实现了回调函数后，接下来定义分组结构体Group：

type Group struct {
	name      string
	getter    Getter //回调函数
	maincache *cache //缓存策略
}

然后初始化，传入回调函数，并且初始化缓存策略结构：

func NewGroup(name string, cacheBytes int64, getter Getter) *Group {
	if getter == nil {
		panic("无回调函数可用来加载缓存")
	}
	mu.Lock()
	defer mu.Unlock()
	g := &Group{
		name:      name,
		getter:    getter,
		maincache: &cache{cacheBytes: cacheBytes},
	}

	groups[name] = g
	return g
}

groups表示分组集合，存放每个分组，mu和groups为全局变量：

//全局变量
var (
	mu     sync.RWMutex
	groups = make(map[string]*Group) //组名字：组
)

然后就可以通过名字来获取对应的分组：

//获得分组
func GetGroup(name string) *Group {
	//读锁，其他写锁需要等待读锁解锁来能进行操作，而其他读锁可以同时操作
	mu.RLock()
	g := groups[name]
	mu.RUnlock()
	return g
}

接下来就是定义获取缓存的方法，根据上面的流程，首先判定缓存是否已经在链表中，若在则直接调取，不在则从其他地方（回调函数或远程节点）获取，然后将缓存存放在链表中。因此先需要一个主函数Get：

//获取缓存
func (g *Group) Get(key string) (ByteView, error) {
	//输入为空
	if key == "" {
		return ByteView{}, fmt.Errorf("key is required")
	}

	if k, ok := g.maincache.get(key); ok {
		log.Println("[GeeCache] hit")
		return k, nil
	}

	//如果在链表中没有找到则启动回调函数来将缓存添加到链表中
	return g.load(key)
}

Get函数判定输入的缓存名字是否为空，且链表中是否存在该缓存，若不存在则调用load函数来加载缓存：

//加载缓存，从远程节点获取或调用回调函数。目前实现回调函数获取
func (g *Group) load(key string) (ByteView, error) {
	//调用回调函数
	return g.getLocally(key)
}

由于没有实现远程节点获取缓存，因此现在调用回调函数来获取：

//封装回调函数
func (g *Group) getLocally(key string) (ByteView, error) {
	//回调函数获取缓存
	bytes, err := g.getter.Get(key)
	if err != nil {
		return ByteView{}, err
	}
	//将bytes数据转换成ByteView
	value := ByteView{b: cloneBytes(bytes)}
	//将该值添加进链表中
	g.populateCache(key, value)
	return value, nil
}

获取之后缓存定义为结构体形式，并调用populateCache函数将缓存存放在链表中，populateCache函数是对添加方法的封装：

//封装添加方法
func (g *Group) populateCache(key string, value ByteView) {
	g.maincache.add(key, value)
}

func TestGet(t *testing.T) {
	loadCount := make(map[string]int, len(db)) //统计加载回调函数的次数，出现两次以上则表示出问题
	gee := NewGroup("scores", 2<<10, GetterFunc(//2<<10，将2往左移动10位，相当于2 * 2^10
		func(key string) ([]byte, error) {
			log.Println("[SlowDB] search key", key)

GeeCache HTTP服务端

服务端解析网址，从网址中解析出分组信息和key，然后从分组中获取对应的缓存并返回到页面中。

首先定义固定地址，包括IP地址和后缀，特定的后缀用于存放缓存。在本例中，后缀为/_geecache/负责存放缓存。因此定义一个结构体来初始化地址：

//定义后缀
const defaultBasePath = "/_geecache/"

type HTTPPool struct {
	self     string //IP地址和端口，例如https://example.net:8000
	basePath string //地址后缀，用于节点间的访问。例如http://example.com/_geecache/
}
//初始化
func NewHTTPPool(self string) *HTTPPool {
	return &HTTPPool{
		self:     self,
		basePath: defaultBasePath,
	}
}

然后定义缓存获取函数：首先将输入格式为/<basepath>/<groupname>/<key>的地址进行分解得到分组名字和key；然后判断输入地址的后缀、分组名字是否有效；随后调用LRU缓存策略，获取对应的缓存；最后将缓存输出到页面中:

//打印服务器信息
func (p *HTTPPool) Log(format string, v ...interface{}) {
	//v的值写入format中
	log.Printf("[server %s]: %s", p.self, fmt.Sprintf(format, v...))
}

//读取缓存并返回缓存
func (p *HTTPPool) ServeHTTP(w http.ResponseWriter, r *http.Request) {
	//判断地址是否属于这一个分支
	if !strings.HasPrefix(r.URL.Path, p.basePath) {
		panic("HTTPPool serving unexpected path: " + r.URL.Path)
	}
	p.Log("%s %s", r.Method, r.URL.Path)
	//分解地址中的参数，地址格式为： /<basepath>/<groupname>/<key> required
	//得到groupname和key
	parts := strings.SplitN(r.URL.Path[len(p.basePath):], "/", 2)
	//判断是否分解成功
	if len(parts) != 2 {
		http.Error(w, "bad request", http.StatusBadRequest)
		return
	}
	groupName := parts[0]
	key := parts[1]

	//通过分组的名字得到分组
	group := GetGroup(groupName)
	if group == nil {
		http.Error(w, "no such group: "+groupName, http.StatusNotFound)
		return
	}
	//通过key获取缓存
	view, err := group.Get(key)
	if err != nil {
		http.Error(w, err.Error(), http.StatusInternalServerError)
		return
	}

	//将缓存发送至页面
	w.Header().Set("Content-Type", "application/octet-stream")
	w.Write(view.ByteSlice())
}

Day4 一致性哈希

分布式的问题

对于分布式缓存来说，当某一节点收到请求后，若本节点没有该请求对应的缓存，则需要从其他节点获取数据或者从数据源获取数据。

而当客户端第二次发出相同的请求后，可能不一定是由之前的节点来接收，这时新的节点需要从其他节点获取数据或者从数据源获取数据，这就导致了不同的节点存储的缓存是相同的且耗时。

解决办法

可采用哈希的方法，给每个节点标志对应的哈希值，并将每个节点放置在一个环上。而对于每个请求key，同样地标志哈希值并放置在环上，然后顺时针查找最近的节点。

若出现增加、删除节点的情况，那么对应的节点上的请求查找下一个相近的节点。

有时服务器的节点过少，因此有些节点会接收大量的请求，而有些节点会接收少量的请求，这时会导致节点间负载不均。

因此引入虚拟节点，即1个真实节点对应多个虚拟节点，并将虚拟节点放置环上，当请求查找到最近的虚拟节点时，则等于查找到真实节点。

一致性哈希的实现

所有节点的哈希值由一个结构体来进行实现并进行存储，存储方式为经排序的数组，并且通过哈希表来存放虚拟节点对应的真实节点，同时需要定义hash函数来为每个节点计算hash值（默认为crc32.ChecksumIEEE算法）:

//自定义hash函数
type Hash func(data []byte) uint32

type Map struct {
	hash     Hash           //哈希函数
	replicas int            //一个真实节点对应的虚拟节点个数
	keys     []int          //环，存放节点的哈希值
	hashMap  map[int]string //键为虚拟节点的hash值，值为对应的真实节点
}

//初始化
func New(replicas int, fn Hash) *Map {
	m := &Map{
		hash:     fn,
		replicas: replicas,
		hashMap:  make(map[int]string),
	}
    if m.hash==nil{
        m.hash=crc32.ChecksumIEEE
    }
	return m
}

增加节点

增加节点，并增加对应的虚拟节点。将这些节点存放在数组中，并对应关系存放在哈希表中，然后对数组排序，方便查找。

//添加节点
func (m *Map) Add(keys ...string) {
	//真实节点个数
	for _, key := range keys {
		//虚拟节点对应的哈希值
		for i := 0; i < m.replicas; i++ {
			//计算哈希值
			hash := int(m.hash([]byte(strconv.Itoa(i) + key))) //将序号转换为string并进行拼接
			m.keys = append(m.keys, hash)                      //将哈希值放置在环上
			m.hashMap[hash] = key                              //将虚拟节点与真实节点对应
		}
	}
	//将数组从小到大排序
	sort.Ints(m.keys)
}

获取节点

通过计算请求节点的哈希值来找到比他大的第一个节点，并返回他的真实节点：

//查找节点
func (m *Map) Get(key string) string {
	if len(m.keys) == 0 {
		return ""
	}
	//计算节点的哈希值
	hash := int(m.hash([]byte(key)))
	//使用二分查找,查找第一个大于等于该哈希值的节点,若没有找到节点则返回数组的长度
	idx := sort.Search(len(m.keys), func(i int) bool {
		return m.keys[i] >= hash
	})
	return m.hashMap[m.keys[idx%len(m.keys)]]
}

Day5 分布式节点

在之前的获取缓存策略中，流程如下：

                            是
接收 key --> 检查是否被缓存 -----> 返回缓存值 ⑴
                |  否                         是
                |-----> 是否应当从远程节点获取 -----> 与远程节点交互 --> 返回缓存值 ⑵
                            |  否
                            |-----> 调用`回调函数`，获取值并添加到缓存 --> 返回缓存值 ⑶

在策略中，已经实现了（1）和（3），本节将实现（2）这一步骤。

问题

  • 远程节点与本地节点的区别

​ 远程节点表示为其他服务器（缓存服务器），而本地节点表示主服务器（用于访问的服务器）。若主服务器与某一缓存服务器一同启动，则本地节点为该缓存服务器。

  • 选择哪个远程节点

    根据key的哈希值来选取远程节点

可将步骤（2）细分为如下流程：

使用一致性哈希选择节点        是                                    是
    |-----> 是否是远程节点 -----> HTTP 客户端访问远程节点 --> 成功？-----> 服务端返回返回值
                    |  否                                    ↓  否
                    |----------------------------> 回退到本地节点处理。

接下来的工作为：

  • 每个客户端建立对应的节点，并通过客户端来输出缓存值。例如客户端http://localhost:8001，通过访问http://localhost:8001/_geecache/groupName/key来得到对应的缓存值。

  • 分组结构中添加节点属性，用于访问节点获取缓存值。

  • 主函数添加api服务器，以便于访问

建立客户端获取接口和缓存值获取接口

在服务端原有的基础上加入节点属性，用该属性来建立节点：

type HTTPPool struct {
	self        string                 //IP地址和端口，例如https://example.net:8000
	basePath    string                 //地址后缀，用于节点间的访问。例如http://example.com/_geecache/
	mu          sync.Mutex             //锁
	peers       *consistanthash.Map    //生成节点
	httpGetters map[string]*httpGetter //存放节点对应的客户端
}

添加节点函数

增加添加节点函数，将客户端存进map中，采用key-客户端映射关系：

//设置节点
func (p *HTTPPool) Set(peers ...string) {
	p.mu.Lock()
	defer p.mu.Unlock()
	//初始化节点
	p.peers = consistanthash.New(defaultReplicas, nil)
	//增加节点
	p.peers.Add(peers...)
	//将节点名字与缓存请求策略绑定
	p.httpGetters = make(map[string]*httpGetter, len(peers))
	for _, peer := range peers {
		//将节点与peer + p.basePath的地址绑定
		p.httpGetters[peer] = &httpGetter{baseURL: peer + p.basePath}
	}
}

查询函数

通过输入的key利用刚建立的map来搜索符合条件的节点客户端：

//根据key来获取节点对应的客户端，并通过客户端来获得缓存
//返回客户端
func (p *HTTPPool) PickPeer(key string) (PeerGetter, bool) {
	p.mu.Lock()
	defer p.mu.Unlock()
	//根据key来获取节点名字
	if peer := p.peers.Get(key); peer != "" && peer != p.self {
		p.Log("Pick peer %s", peer)
		return p.httpGetters[peer], true
	}
	return nil, false
}

Day6 防止缓存击穿

缓存雪崩、缓存击穿、缓存穿透

缓存雪崩：指的是缓存同一时间失效，导致DB请求量剧增、压力骤增。
缓存击穿：指的是当缓存失效的同时有大量的请求
缓存穿透：指的是大量请求不存在的缓存

创建锁结构

创建两个锁结构，一个锁用于全局控制，一个锁用于局部控制。

//锁结构，并存储不同key的请求
type Group struct {
	mu sync.Mutex
	m  map[string]*call
}

//key对应的值
type call struct {
	wg  sync.WaitGroup
	val interface{}
	err error
}

Group结构用于存储key-值的映射关系，而值的修改和存储由call结构来实现。sync.WaitGroup用于控制并发，它的使用方法有：

  1. main协程通过调用 wg.Add(delta int) 设置worker协程的个数，然后创建worker协程；
  2. worker协程执行结束以后，都要调用 wg.Done()；
  3. main协程调用 wg.Wait() 且被block，直到所有worker协程全部执行结束后返回。

为Group建立方法，该方法通过初始化map，将key-cal对象一一对应：

/*
避免同一时间创建对象来访问同一缓存
设置多个锁的好处在于，多个线程能在删除字典之前同时进入函数中来取值，提高了并发效率
*/
func (g *Group) Do(key string, fn func() (interface{}, error)) (interface{}, error) {
	//上锁
	g.mu.Lock()
	if g.m == nil {
		g.m = make(map[string]*call)
	}
	if c, ok := g.m[key]; ok {
		g.mu.Unlock()
		//其他线程在这儿等待赋值成功
		c.wg.Wait()
		return c.val, c.err
	}
	c := new(call)
	c.wg.Add(1)
	g.m[key] = c
	//初始化c后解锁
	g.mu.Unlock()

	//赋值
	c.val, c.err = fn()
	c.wg.Done()

	//删除键值对
	g.mu.Lock()
	delete(g.m, key)
	g.mu.Unlock()
	return c.val, c.err
}

该方法的作用为：首先上锁，避免多个协程初始化或修改map，而后解锁使得多个协程能够访问map，但此时并没有给call结构赋值，因此通过call结构的锁来限制协程访问值，于是有了sync.WaitGroup的三个方法:Add()、Done()、Wait()。使用wait方法来阻塞协程，通过Add增加协程运作数，调用Done()来表示协程运行完毕，协程运行完毕后被阻塞的协程开始运行。在Do方法中，利用WaitGroup来将协程阻塞在读取call结构里的值这一步，当某个协程赋值完成后其他协程开始读取值并返回值，而在读取之后删除map中的键值对来更新缓存，也能够节省内存。

然后在读取缓存的Group中加入该锁的机制，引入group结构：

type Group struct {
	name      string
	getter    Getter              //回调函数
	maincache *cache              //缓存策略
	peers     PeerPicker          //加载节点
	loader    *singleflight.Group //避免同一时间发出请求，提高并发效率
}

load函数用Do函数来将原来的逻辑包裹起来：

//加载缓存，从远程节点获取或调用回调函数。目前实现回调函数获取
func (g *Group) load(key string) (ByteView, error) {
	//采取锁机制，避免同一时间访问
	viewi, err := g.loader.Do(key, func() (interface{}, error) {
		//调用远程节点
		if g.peers != nil {
			//获取节点对应的客户端
			if peer, ok := g.peers.PickPeer(key); ok {
				//从客户端得到缓存
				if value, err := g.getFromGetter(peer, key); err == nil {
					return value, nil
				}
				log.Println("[GeeCache] Failed to get from peer", peer)
			}
		}
		//调用回调函数
		return g.getLocally(key)
	})
	if err == nil {
		return viewi.(ByteView), nil
	}
	return viewi.(ByteView), err
}
